# 高端智能多功能点单管理系统——核心架构与控制逻辑技术文档

本工程基于 Qt 框架开发，采用改良型的 **MVC（Model-View-Controller）** 架构设计。系统深度整合了电子菜单展示、会员动态无感算价、购物车多级联动、订单批量结算以及多线程仿真排队看板等核心业务。

以下为本系统的架构设计思路与控制逻辑的深度解析。

---

## 一、 系统核心架构设计思路

系统通过分层架构实现了“高内聚、低耦合”的设计目标，整体架构分为三层：**视图层 (View)**、**核心控制引擎 (Controller)** 和 **数据实体/管理模型层 (Model)**。

### 1. 数据模型层 (Model)——单例资源管理器

模型层由 `DishManager`、`MemberManager`、`OrderManager` 和 `QueueManager` 组成。

* **设计模式**：所有管理器均采用  Meyers 单例模式 (`instance()`)，确保在整个应用程序生命周期内，全局业务数据（如订单历史、排队序列、会员积分）有且仅有一个信任源，避免多实例导致的数据不一致。
* **内存隔离**：模型层完全不包含任何 Qt 界面组件，仅负责纯粹的数据结构维护（如 `QList<DishModel>`、`QList<QueueNode>`），以便于进行单元测试或未来向非 GUI 环境迁移。

### 2. 控制联动层 (Controller)——业务中枢

* **核心组件**：`SystemEngine` 类充当整个系统的业务联动引擎。
* **桥接机制**：视图层不直接跨层访问底层 Model，而是将所有用户交互转化为对 `SystemEngine` 标准 API 的调用。`SystemEngine` 负责编排跨模型的复杂组合业务。例如：下单时，它需要协同 `DishManager` 查询基价、协同 `MemberManager` 计算折扣并累加积分、最后协同 `OrderManager` 落地订单。

### 3. 视图展现层 (View)——数据驱动界面

* **核心组件**：`MainWindow` 与 `Ui_MainWindow`。
* **无状态驱动**：界面内部不持久化业务核心数据，仅维护一个轻量级的购物车缓冲区 `QMap<QString, int> m_cart`（键：菜品名，值：数量）。界面的动态刷新完全依赖于控制器的信号与槽（Signals and Slots）机制。

## 二、 核心控制逻辑与联动流程

### 1. 购物车动态多级算价与会员无感登录机制

当用户在 `tableWidgetMenu` 中双击菜品或点击“加入购物车”按钮时，系统将触发级联计算：

```
[用户动作: 添加菜品/输入卡号]
       │
       ▼
MainWindow::recalculateCartPrices() 
       │
       ├─► 遍历购物车缓冲 m_cart 
       │
       ├─► 调用 SystemEngine::calculatePrice(dishName, memberId)
       │         │
       │         ├─► MemberManager::getDiscount(memberId) ──► 根据积分梯度映射折扣 (95折/9折/85折)
       │         └─► 返回折后单价
       │
       ▼
[计算合流] ──► 动态更新 QTableWidgetCart 每一行的小计
       │
       └─► 刷新 lblOriginalPrice(原价总额) / lblDiscountPrice(减免额) / lblFinalPrice(实付)

```

* **无感登录控制**：`lineEditMemberId` 绑定了 `textChanged` 信号。当用户输入卡号时，无需点击“登录”按钮，程序自动逆向探测其会员等级并重新触发 `recalculateCartPrices()`，实现界面价格的实时无缝刷新。

### 2. 订单批量提交与排队看板自动化连锁控制

这是本系统最核心的**复合业务控制流**，位于 `MainWindow::on_btnSubmitOrder_clicked()` 中：

```cpp
void MainWindow::on_btnSubmitOrder_clicked() {
    // 1. 防御性编程：校验购物车是否为空
    if (m_cart.isEmpty()) { ... return; }

    QString mid = ui->lineEditMemberId->text().trimmed();
    // 自动化凭证分流：未登录散客生成“客流水-XXX”，会员直接使用会员号
    QString queueIdentity = mid.isEmpty() ? QString("客流水-%1").arg(qrand() % 900 + 100) : mid;

    // 2. 事务性原子提交：遍历购物车，批量向底层模型写入真实多菜品账单
    QMap<QString, int>::const_iterator it = m_cart.constBegin();
    while (it != m_cart.constEnd()) {
        for (int i = 0; i < it.value(); ++i) {
            m_engine->createOrder(it.key(), mid); // 联动累加会员积分，发射 menuDataChanged 信号
        }
        it++;
    }

    // 3. 自动化连锁：触发自动排队业务，默认加入“现场取餐进程” (type == 1)
    int defaultQueueTab = 1; 
    m_engine->customerJoinQueue(queueIdentity, defaultQueueTab); // 发射 queueStatusChanged 信号

    // 4. 清理现场
    m_cart.clear();
    recalculateCartPrices();
    loadHistoryOrders(); // 刷新点餐记忆历史

    // 5. 视图路由重定向：自动引导切换至排队界面看板
    ui->tabWidgetQueueType->setCurrentIndex(defaultQueueTab);
    ui->sidebarNav->setCurrentRow(2); 
}

```

### 3. 队列消解仿真控制逻辑

为了模拟真实餐厅中前排顾客不断取餐离开的动态场景，系统在 `QueueManager::getQueue(int type)` 中设计了**初次加载保持与后续刷新递减**的仿真逻辑：

```cpp
QList<QueueNode> QueueManager::getQueue(int type) {
    QList<QueueNode> *targetQueue = (type == 0) ? &m_reserveQueue : &m_pickupQueue;
    bool *isFirstCall = (type == 0) ? &m_reserveFirstCall : &m_pickupFirstCall;

    // 核心仿真机制
    if (*isFirstCall) {
        *isFirstCall = false; // 消费掉首次加载标记，保证初始化时 10 名模拟伪用户正常呈现
    } else {
        // 之后每调用（刷新）一次，自动从队列头部移除 2 人
        for (int i = 0; i < 2; ++i) {
            if (!targetQueue->isEmpty()) {
                targetQueue->removeFirst(); // 模拟前排客户已完成业务
            }
        }
    }
    return *targetQueue;
}

```

* **控制效果**：当用户在界面上点击 `btnRefreshQueue`（刷新排队进度）或切换排队 Tab 时，由于触发了该函数的重新调用，排队网格中的等待人数会自动化递减 2 人，精准模拟了业务消解过程。

### 4. 历史订单克隆重现控制

系统支持从过往消费记录中快速一键“再来一单”，其控制逻辑利用了正则表达式进行精准的反向解析：

1. 用户在历史列表中选中某行文本（标准格式如：`订单:1001 | 菜品: 招牌红烧肉`）。
2. 点击“复制该订单再来一单”，触发 `OrderManager::duplicateFromText`。
3. 后端使用非贪婪匹配正则表达式 `菜品:\s*(.*)` 精准提取菜品字符串。
4. 解析成功后直接调用 `addOrder(extractedDishName)` 快捷复制下单，并向全系统广播信号更新 UI。

## 三、 系统对象通信与数据同步机制

系统采用 Qt 的核心基石 **元对象系统（Meta-Object System）** 进行异步或同步通信，完全断开了视图对模型的直接持有。

### 1. 信号广播拓扑关系

在 `SystemEngine` 中定义了两个关键的全局状态信号：

* `signals: void queueStatusChanged();`
* `signals: void menuDataChanged();`

在 `MainWindow::initUIStyleAndConnections()` 中建立监听：

```cpp
connect(m_engine, &SystemEngine::queueStatusChanged, this, &MainWindow::refreshQueueUI);
connect(m_engine, &SystemEngine::menuDataChanged, this, &MainWindow::refreshDishMenuUI);

```

### 2. 经典数据流同步场景

当用户点击“确认付款”时，控制流进入 `SystemEngine::createOrder`。该函数向底层 `MemberManager` 追加积分后，触发 `emit menuDataChanged();`。
此时，订阅了该信号的 `MainWindow::refreshDishMenuUI()` 立即被激活，使电子菜单中可能发生变更的状态或爆款标记得到即时冲刷同步。同理，排队序列的任何变更都会引发 `queueStatusChanged` 的广播，使全界面的排队计数器和顺位表格得到自动化同步更新。
