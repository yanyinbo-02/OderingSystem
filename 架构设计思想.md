本工程是一个基于 Qt 框架、采用 CMake 构建系统的典型 **MVC（Model-View-Controller，模型-视图-控制器）** 软件架构项目。其核心设计哲学是**关注点分离（Separation of Concerns, SoC）**，通过将数据、业务逻辑和用户界面强解耦，确保系统具备高内聚、低耦合的工程特性。

---

## 一、 整体层级架构概况

项目的目录结构按功能职责被划分为四个清晰的层级，形成了自下而上的依赖关系：

```
OrderingSystem (项目根目录)
├── .vscode/               # 编译器与开发环境本地配置层
├── build/                 # 编译产物层（隔离源码与中间件）
├── resources/             # 静态资源层（图标、样式表、图片等）
└── src/                   # 核心源码层
    ├── models/            # 1. 模型层 (Data Layer)：纯数据结构与原子操作
    ├── controllers/       # 2. 控制层 (Logic Layer)：业务逻辑与数据中枢
    └── views/             # 3. 视图层 (Presentation Layer)：UI 渲染与交互

```

### 数据与控制流向原则

1. **单向依赖**：`Views` 依赖 `Controllers` 提供的接口，`Controllers` 操纵 `Models`。`Models` 绝对不感知 `Views` 的存在。
2. **事件驱动**：`Views` 通过 Qt 的信号与槽（Signals & Slots）机制向 `Controllers` 异步传递用户指令；`Controllers` 状态改变后通过信号通知 `Views` 刷新，避免阻塞主线程。

## 二、 核心层级详细阐述

### 1. 模型层 (Models Layer)

* **职责**：负责系统核心实体数据的定义、封装与持久化抽象。该层文件不包含任何 Qt 界面组件（如 `QWidget`、`QMessageBox`），仅允许使用 Qt 的基础容器类（如 `QString`、`QVector`）和标准 C++ 结构。
* **设计解耦**：模型类应是“哑模型（Dumb Models）”，只暴露出 Getter/Setter 接口及基础的序列化逻辑，不参与复杂的跨对象业务计算。

### 2. 控制层 (Controllers Layer)

* **职责**：整个系统的“大脑”与调度中枢。它负责协调各个 Model 之间的交互，执行核心业务算法（如：会员折扣矩阵计算、Top 5 菜品多级排序、订单的深拷贝复制、排队队列的 FIFO 管理）。
* **架构模式**：通常采用**中介者模式（Mediator Pattern）**或**单例模式（Singleton Pattern）**。它向 View 层隐蔽了底层复杂的数据结构，View 层只需调用其高层业务接口。

### 3. 视图层 (Views Layer)

* **职责**：负责用户界面的布局、绘制与人机交互事件的捕获。
* **组件化设计**：包含主窗口与复合控件。通过将复杂的界面拆分为 `widgets/` 目录下的独立小组件（如自定义菜品卡片、排队看板单元），实现 UI 代码的复用并降低单个文件的维护难度。

## 三、 具体文件职责逐一拆解

### 1. 环境与构建配置文件

* **`CMakeLists.txt`**
* **职责**：跨平台构建脚本。定义了 C++ 编译标准（如 C++17/C++20）、自动调用 Qt 的元对象编译器（MOC、UIC、RCC）、指定头文件搜索路径、将 `src/` 下的所有源码编译链接为最终的可执行程序。


* **`.vscode/settings.json`**
* **职责**：IDE 本地开发环境配置文件。如代码内注释所示，用于重置 `cmake.configureArgs` 传参，确保 VS Code 的 CMake Tools 插件完全遵循 `CMakeLists.txt` 的原生指示，防止本地缓存污染导致编译失败。



### 2. 核心源码文件 (`src/`)

* **`main.cpp`**
* **职责**：应用程序的唯一入口点。初始化 `QApplication` 生命周期主循环，实例化全局控制中枢（`SystemEngine`）和主视窗（`MainWindow`），通过信号与槽建立最初的骨架连接，最后展示界面。



#### 【Models 文件夹】

* **`dish.h` / `dish.cpp**`
* **职责**：菜品实体类。定义单个菜品的属性（ID、名称、单价、分类、描述、销量、评价集合）。提供基础的原子操作，如添加单条评价、更新累计销量。


* **`member.h` / `member.cpp**`
* **职责**：会员实体类。维护会员的基础身份信息、积分总额及会员等级。包含积分累加逻辑，以及根据积分阶梯触发会员等级晋升的条件断言。


* **`order.h` / `order.cpp**`
* **职责**：订单实体类。封装单次交易的数据，包含订单流水号、时间戳、购买者（会员/散客）、商品条目容器（`QList<QPair<Dish*, int>>`）以及最终结算总价。


* **`queue.h` / `queue.cpp**`
* **职责**：排队实体类。封装底层队列结构（如 `QQueue`），分别管理“预约等位队列”和“后厨取餐队列”的数据状态，提供入队、出队、获取当前排队进度的原子接口。



#### 【Controllers 文件夹】

* **`system_engine.h` / `system_engine.cpp**`
* **职责**：全局核心控制器（System Controller）。
* **数据持有**：维护全局的菜品主表、会员数据库、活动订单表及当前排队队列。
* **业务逻辑实现**：响应 View 层的请求，在此处集中实现项目要求的功能二（折扣积分算法）、功能三（双关键字排序）、功能四（Top 5 推荐权重算法）及功能五（订单深拷贝）。



#### 【Views 文件夹】

* **`mainwindow.ui`**
* **职责**：XML 格式的界面布局文件。由 Qt Designer 生成，直观定义主界面的基础网格布局（Grid Layout）、菜单栏、工具栏及主要视窗控制器的摆放位置。


* **`mainwindow.h` / `mainwindow.cpp**`
* **职责**：主窗口类。承载系统的主要视觉呈现，将 UI 上的按钮点击、表格选择等事件转化为对 `SystemEngine` 的业务调用；同时监听 `SystemEngine` 抛出的数据变更信号，动态更新界面。


* **`widgets/` (文件夹)**
* **职责**：高内聚自定义微控件的存放地。例如，为了满足功能一的菜单分类展示，可在此设计 `DishCardWidget`（菜品卡片控件）；为了实现功能六的排队刷新，可在此设计 `QueueStatusWidget`（排队进度看板）。
